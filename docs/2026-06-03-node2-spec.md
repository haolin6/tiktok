# 2026-06-03 节点 2 详细 Spec：P0 主链路打通

## 1. 今日目标

节点 2 只证明 P0 主链路已经能完整演示：

后台创建商品和 30 秒竞拍 -> 用户进入直播间 -> 用户出价 -> 价格和领先者更新 -> 竞拍结束 -> 自动成交或流拍 -> 生成订单 -> 用户模拟支付。

本节点可以先用 HTTP 轮询或手动刷新保证主链路，不强制实现 WebSocket、排行榜动画、100 人并发脚本、Redis 幂等和真实支付。这些留到 2026-06-06 节点 3。

## 5. 任务 2：出价 API

完成内容：

- 新增 `POST /api/auctions/:id/start`。
  - 将 `Scheduled` 竞拍推进到 `Running`。
  - 用于管理端创建竞拍后立即开始 30 秒演示。
  - 写入 `auction_events`：`auction.started`。
- 新增 `POST /api/auctions/:id/bids`。
  - 请求体：`userId`、`amount`、`requestId`。
  - 校验竞拍存在。
  - 校验用户存在且是 `bidder`。
  - 校验竞拍状态是 `Running`。
  - 校验当前服务端时间未超过 `endAt`。
  - 校验金额必须等于下一口价：`currentPrice + incrementStep`；如果封顶价剩余不足一个加价幅度，则下一口价等于 `ceilingPrice`。
  - 出价成功后写入 `bids`，`accepted = true`。
  - 更新 `auctions.current_price`、`auctions.current_winner_id`、`auctions.version`。
  - 写入 `auction_events`：`bid.accepted`。
  - 返回最新 snapshot。
- 对无效出价写入 `bids`，`accepted = false`，`reject_reason` 记录失败原因。
  - 金额过低或不符合下一口价返回 409。
  - 竞拍未开始、已结束、已成交、已流拍、已取消返回 409。
  - 用户不存在或不是出价用户返回 400。
  - `requestId` 重复不能再次抬价，返回 409 或返回原拒绝原因，但必须保证价格不变。

约束：

- 今天先用 MySQL 事务和行锁保证单机一致性；不强制接 Redis 锁。
- 出价事务内必须锁定目标 auction 行，避免两个并发请求读到同一个当前价。
- `requestId` 使用现有唯一键 `uniq_bids_request (auction_id, user_id, request_id)` 做幂等兜底。
- 不允许为了让测试通过而放宽金额校验。
- 失败响应必须是统一格式：`{ error: { code, message } }`。
- 出价成功和拒绝都要能在 MySQL 中追踪，方便演示“系统不是只改了前端”。

测试：

- 集成测试覆盖一次合法出价，断言：
  - HTTP 200。
  - `auctions.current_price` 更新为下一口价。
  - `auctions.current_winner_id` 更新为出价用户。
  - `bids.accepted = true`。
  - `auction_events` 有 `bid.accepted`。
- 集成测试覆盖金额过低、金额跳档、非 bidder 用户、终态竞拍出价。
- 集成测试覆盖重复 `requestId` 不会导致价格重复上涨。

## 6. 任务 3：成交、流拍和订单生成

完成内容：

- 新增竞拍结算逻辑 `settleAuctionIfNeeded` 或同等 service 方法。
- 出价达到 `ceilingPrice` 时立即成交：
  - `Running -> Sold`。
  - 创建 `orders`，状态为 `pending_payment`。
  - 写入 `auction_events`：`auction.sold`、`order.created`。
  - 返回 snapshot 时带上订单信息。
- 竞拍到 `endAt` 后自动结算：
  - 如果存在 `current_winner_id`，则 `Running -> Sold` 并生成订单。
  - 如果没有有效出价，则 `Running -> Passed`，不生成订单。
  - 写入 `auction_events`：`auction.sold` 或 `auction.passed`。
- 结算触发方式：
  - `GET /api/auctions/:id/snapshot` 调用时先检查是否到期。
  - `POST /api/auctions/:id/bids` 调用时如果已到期，先结算再拒绝出价。
  - `GET /api/orders` 调用时可以补结算已到期的 Running 竞拍。

约束：

- 今天不引入后台定时器、队列或 cron；使用“按需自动结算”即可满足演示闭环。
- 订单必须依赖 `orders.auction_id` 唯一键，确保一个竞拍最多只有一个订单。
- 如果订单已存在，重复结算不能创建第二个订单。
- `Sold`、`Passed`、`Canceled` 结算后不可逆。
- 取消竞拍后不允许生成订单。
- 结算逻辑必须和出价逻辑共用状态机，不允许直接硬改终态。

测试：

- 集成测试覆盖封顶价立即成交并生成订单。
- 集成测试覆盖到期后最高出价人成交并生成订单。
- 集成测试覆盖无有效出价到期后流拍，不生成订单。
- 集成测试覆盖重复调用 snapshot 或 orders 不会重复生成订单。

## 12. 最终手工验收

启动服务：

```bash
npm run dev
```

验收步骤：

1. 打开管理端 `/admin/auctions/new`。
2. 创建一个 30 秒竞拍，起拍价例如 99，加价幅度 10，封顶价 129。
3. 开始竞拍。
4. 打开用户端 `/live/:roomId`。
5. 选择用户 A 出价，页面价格从 99 变为 109。
6. 切换用户 B 出价，页面价格从 109 变为 119。
7. 再出价达到 129，竞拍立即变成 `Sold`。
8. 打开 `/admin/orders`，能看到该成交订单。
9. 成交用户打开 `/pay/:orderId`，点击模拟支付。
10. 订单状态变成 `paid`。

如果不用封顶价立即成交，也必须跑通到期成交：

1. 创建一个 30 秒竞拍。
2. 至少一个用户出价成功。
3. 等倒计时结束。
4. 刷新直播间或订单列表。
5. 竞拍变成 `Sold`，并生成一条订单。

MySQL 验收查询：

```sql
SELECT id, status, current_price, current_winner_id, version
FROM auctions
WHERE id = <auction_id>;

SELECT auction_id, user_id, amount, request_id, accepted, reject_reason
FROM bids
WHERE auction_id = <auction_id>
ORDER BY id;

SELECT id, auction_id, buyer_id, amount, status
FROM orders
WHERE auction_id = <auction_id>;

SELECT event_type, created_at
FROM auction_events
WHERE auction_id = <auction_id>
ORDER BY id;
```

通过标准：

- HTTP 页面操作能跑完整路径。
- MySQL 中能看到竞拍、出价、订单和事件日志。
- 自动测试和构建通过。
- 即使暂时没有 WebSocket，也必须能从后台发布走到用户成交订单。

## 14. 来不及时的砍范围规则

保留：

- 后台能创建竞拍。
- 用户能进入直播间。
- 用户能出价。
- MySQL 能记录出价。
- 结束后能生成订单。
- 管理端能看到订单。
- 用户能模拟支付。

可砍：

- 漂亮后台样式。
- 历史记录复杂筛选。
- 最近出价列表的动画。
- 自动轮询频率优化。
- 移动端细节打磨。

最低通过线：

后台发布 -> 用户出价 -> 成交/流拍 -> 订单生成，这条链路必须真实写入 MySQL，不能只靠前端假数据。
