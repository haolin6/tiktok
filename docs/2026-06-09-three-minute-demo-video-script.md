# 2026-06-09 三分钟成果演示视频脚本

本脚本用于录制约 3 分钟公开视频或提交页演示视频。目标是快速证明：

```text
后台发布竞拍 -> 用户直播间实时出价 -> 管理端取消 -> 成交订单 -> 模拟支付 -> 并发一致性可解释
```

默认本地地址：

- Web：`http://127.0.0.1:3000`
- API：`http://127.0.0.1:4000`

录制前准备：

```bash
npm run check:env
npm run db:migrate
npm run db:seed
npm run dev
```

录屏前建议准备两个浏览器窗口：

- 窗口 1：后台 `/admin/auctions/new`。
- 窗口 2 和窗口 3：同一个 `/live/:roomId`，分别选择 `竞拍用户 A` 和 `竞拍用户 B`。

## 时间轴

### 0:00-0:15 项目一句话

画面：

- 展示 `README.md` 顶部或 HTML PPT 封面。

口播：

> 这是一个抖音电商直播竞拍全栈系统。当前版本用固定视频模拟直播间，重点展示后台发布竞拍、用户实时出价、WebSocket 同步、成交订单、模拟支付，以及 Redis 和 MySQL 保护下的并发一致性。

必须说清楚：

- 当前不是完整真实直播推流。
- 当前不是接入真实支付。

### 0:15-0:45 后台发布竞拍

页面：

- `/admin/auctions/new`

操作：

1. 填写或保留默认商品信息。
2. 起拍价建议 `99`。
3. 加价幅度建议 `10`。
4. 封顶价建议 `129`，便于快速成交。
5. 时长建议 `30` 秒。
6. 点击 `创建并开始`。

画面成功点：

- 页面显示竞拍 ID。
- 状态为 `Running`。
- 出现 `进入直播间` 链接。

口播：

> 主播在后台配置商品和竞拍规则，包括起拍价、加价幅度、封顶价和竞拍时长。创建后系统进入 Running 状态，用户端可以进入对应直播间参与出价。

### 0:45-1:25 双端实时出价

页面：

- 两个 `/live/:roomId` 窗口并排。

操作：

1. 左侧选择 `竞拍用户 A`。
2. 右侧选择 `竞拍用户 B`。
3. 用户 A 点击出价。
4. 用户 B 看到当前价、领先者、最近出价和排行榜变化。
5. 用户 B 再点击出价。
6. 用户 A 看到同步变化。

画面成功点：

- 连接状态为 `connected`。
- 当前价从 `¥99.00` 变为 `¥109.00`，再变为 `¥119.00`。
- 领先者从用户 A 变为用户 B。

口播：

> 直播间通过 Socket.IO 做房间级广播。一个用户出价后，另一个窗口不需要刷新，就能看到价格、领先者和排行榜同步更新。

### 1:25-1:50 管理端取消竞拍

页面：

- `/admin/auctions`

操作：

1. 准备一个非终态竞拍，状态为 `Scheduled` 或 `Running`。
2. 点击该行 `取消`。
3. 在弹窗中输入取消原因，例如 `demo cancel`。
4. 确认后观察列表状态。

画面成功点：

- 竞拍状态变为 `已取消`。
- 如果直播间正在订阅该竞拍，直播间可显示 `竞拍已取消`。

口播：

> 取消能力现在可以直接从管理端列表触发。后端仍然复用原来的取消 API 和状态机，终态竞拍不会显示取消按钮，也不能再次取消。

注意：

- 如果刚才的竞拍已经封顶成交，就新建一个短竞拍用于取消，不要对 `Sold` 状态强行取消。

### 1:50-2:25 成交、订单和模拟支付

页面：

- `/live/:roomId`
- `/pay/:orderId`
- `/admin/orders`

操作：

1. 继续使用另一个未取消竞拍。
2. 用户继续出价直到达到封顶价，或等待到期成交。
3. 赢家点击 `去支付`。
4. 在 `/pay/:orderId` 点击 `模拟支付`。
5. 打开 `/admin/orders` 查看订单状态。

画面成功点：

- 直播间显示 `成交`。
- 支付页状态从 `pending_payment` 变为 `paid`。
- 后台订单列表显示对应订单。

口播：

> 出价达到封顶价后，系统自动推进到成交状态并生成订单。这里是模拟支付，不接真实支付渠道，但订单状态、买家、金额都会写入 MySQL。

### 2:25-2:50 并发一致性

页面：

- 终端。

操作：

```bash
npm run demo:concurrency -- --mode=unique
```

画面成功点：

重点保留以下输出：

```text
attempts=100
accepted=1
rejected=99
finalStatus=Sold
orderCount=1
acceptedBidRows=1
rejectedBidRows=99
```

口播：

> 这段脚本模拟 100 个用户同时抢同一口价。最终只有一个请求成功，只有一个订单生成。Redis lock 保护并发进入关键区，Redis 幂等 key 处理重复请求，MySQL 唯一约束兜底订单唯一性。

如果时间不够：

- 只录 `unique` 模式。
- `duplicate-accepted`、`duplicate-rejected`、`lock-busy` 三种模式放在 README 或答辩文档中说明。

### 2:50-3:00 收束

画面：

- 展示 README 的技术栈和边界，或展示 HTML PPT 的架构页。

口播：

> 当前版本的重点是完整主链路、实时同步和并发一致性证明。真实直播流、真实支付、复杂鉴权和千级线上压测没有包装成已完成能力，而是作为后续扩展方向。

## 录制注意事项

- 视频控制在 3 分钟左右，不要录完整 10-15 分钟答辩流程。
- 画面优先展示页面变化，其次展示终端证据。
- 不要口播“线上 Demo 已部署”“真实支付已接入”“真实直播已接入”。
- 如果现场实时事件偶发延迟，可以刷新直播间展示最新 snapshot，但要说明权威状态来自后端和 MySQL。
- 如果并发命令受本地环境影响失败，不要用旧输出冒充新结果；可以改录 README 中的并发说明，并在提交说明里标注本次视频未现场跑并发。

## 最终视频素材检查

录完后检查视频中是否出现：

- 项目名称和一句话定位。
- 后台发布竞拍页面。
- 双端直播间实时出价同步。
- 管理端取消按钮和 `已取消` 状态。
- 成交结果、模拟支付和后台订单。
- 至少一个并发一致性证据画面。
- 当前未实现边界说明。
