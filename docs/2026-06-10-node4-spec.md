# 2026-06-10 节点 4 详细 Spec：冻结版本 + 演示材料 + 最终验收

## 1. 节点目标

节点 4 的目标不是继续堆功能，而是把已经完成的直播竞拍系统冻结成一个可提交、可录屏、可答辩的版本。

最终必须能证明：

```text
后台发布竞拍 -> 用户进入直播间 -> 双端实时出价 -> 自动延时/成交/取消 -> 订单生成 -> 模拟支付 -> 后台和用户端可查看结果 -> 并发一致性可解释
```

本节点只允许修复影响演示和验收的 bug，补齐文档、演示脚本、录屏清单和最终测试证据。不新增真实直播流、复杂鉴权、复杂看板、大规模压测或数据库大改。

## 2. 依据和前置条件

依据：

- `docs/development_schedule.md` 节点 4。
- `docs/requirements_and_architecture.md` 的演示路径和交付物清单。
- `docs/2026-06-06-node3-spec.md` 的进入节点 4 条件。
- `docs/2026-06-06-node3-test-supplement-spec.md` 的实时和并发补充测试要求。
- `docs/websocket_event_protocol.md`。
- `docs/concurrency_demo.md`。

进入节点 4 前必须保留的能力：

- 后台可以创建、启动、查看竞拍。
- 用户端直播间可以出价并看到当前价、领先者、倒计时和结果。
- Socket.IO 可以按直播间和竞拍隔离广播。
- Redis 锁和幂等 key 可以保护并发出价。
- MySQL 唯一键和事务可以兜底订单唯一性。
- Playwright E2E 可以证明双页面实时链路。

## 3. 修改边界

允许修改：

- `docs/**`：补节点 4 spec、演示脚本、答辩提纲、最终验收清单、AI 使用说明、架构/状态机/数据库说明。
- `scripts/**`：只允许新增或调整非破坏性的验收辅助脚本。
- `package.json`：只允许新增最终验收脚本，不能破坏已有命令。
- `apps/**`、`packages/**`：只允许修复 P0 演示路径上的 bug、明显文案错误、测试稳定性问题或缺失的 `data-testid`。

禁止修改：

- 不新增真实直播推流。
- 不新增完整登录鉴权、网关、限流。
- 不大改数据库表结构。
- 不重构前后端架构。
- 不把 P1/P2 展示效果变成 P0 阻塞项。
- 不提交真实 API Key、个人凭据或本地私密 `.env`。

## 4. P0 任务

### 任务 1：冻结功能基线

完成内容：

- 重新启动 API 和 Web 服务后，管理端和用户端都能打开。
- 后台发布竞拍稳定。
- 用户端出价稳定。
- WebSocket 同步稳定。
- 自动延时、成交、取消、订单、模拟支付稳定。

验收方法：

- `npm run check:env`
- `npm run redis:ping`
- `npm run db:mysql:ping`
- `npm run test`
- `npm run build`
- `npm run test:e2e`

验收标准：

- 所有命令通过。
- 如果某条命令失败，必须明确是环境问题、测试问题还是业务回归。
- 业务回归必须在进入最终提交前修复。

边界条件：

- `npm run test` 不能只证明 HTTP happy path，还必须覆盖拒绝出价、重复请求、自动延时、终态结算、WebSocket 隔离和 Redis 幂等。
- `npm run test:e2e` 至少证明两个页面通过实时事件同步，并走到成交和支付。

### 任务 2：完整演示脚本

完成内容：

- 写出 10-15 分钟演示顺序。
- 每一步必须包含页面路径、操作、预期现象、备用路线。
- 明确演示身份：主播/后台、用户 A、用户 B。
- 明确哪些操作需要提前准备，哪些必须现场演示。

建议演示顺序：

1. 业务场景和目标。
2. 后台发布竞拍。
3. 两个用户进入直播间。
4. 用户 A/B 交替出价，观察 WebSocket 同步。
5. 触发自动延时。
6. 触发封顶成交或倒计时成交。
7. 查看订单并模拟支付。
8. 后台查看竞拍和订单。
9. 运行并发脚本。
10. 讲架构、状态机、Redis/MySQL 分工和 AI 使用。

验收标准：

- 任何人照着脚本可以完成录屏。
- 每一步都有“成功看什么”和“失败怎么兜底”。

边界条件：

- 如果实时事件现场不稳定，备用路线是刷新 snapshot 并展示 MySQL/事件表证据。
- 如果并发现场不稳定，备用路线是展示已保存的并发输出和 MySQL 查询证据。

### 任务 3：技术文档补齐

完成内容：

- 架构图：前端、API、Socket.IO、Redis、MySQL、脚本的关系。
- 状态机图：Draft/Scheduled/Running/Sold/Passed/Canceled 的合法流转。
- 数据库表说明：核心 7 张表的职责、关键字段、关键约束。
- WebSocket 事件协议索引：引用 `docs/websocket_event_protocol.md`，说明房间隔离和重连恢复。
- 并发一致性设计：Redis lock、幂等 key、MySQL unique/order transaction 的三层保护。
- AI 使用说明：AI 做了什么、人审核了什么、如何追溯 spec/测试/修复。

验收标准：

- 答辩时能用 2 分钟讲清架构。
- 能说明为什么不会重复成交、不会重复支付、不会跨房间串消息。
- 文档和当前代码命名一致。

边界条件：

- 不写“未来会做”的大饼当成交付能力。
- 不把未验证的性能数字写成已验证结论。

### 任务 4：并发验收证据

完成内容：

- 保留唯一请求 100 人并发演示。
- 保留重复 accepted 请求演示。
- 保留重复 rejected 请求演示。
- 保留 lock-busy 演示。
- 文档中说明如何用 MySQL 验证订单唯一性和事件行。

验收命令：

```bash
npm run demo:concurrency -- --mode=unique
npm run demo:concurrency -- --mode=duplicate-accepted
npm run demo:concurrency -- --mode=duplicate-rejected
npm run demo:concurrency -- --mode=lock-busy
```

验收标准：

- `orderCount` 必须符合脚本模式预期。
- `acceptedBidRows`、`rejectedBidRows`、`lockBusyBidRows` 与脚本输出一致。
- `auction_events` 能看到 `bid.accepted`、`bid.rejected`、`auction.sold`、`order.created` 等关键事件。

边界条件：

- 并发脚本不能清空业务表。
- 脚本必须创建独立测试商品、竞拍和用户，避免污染演示数据。
- `--mode` 参数必须真实传入内部脚本。

### 任务 5：录屏材料清单

完成内容：

- 双端功能演示录屏清单。
- 并发测试演示录屏清单。
- 关键代码/架构解释录屏清单。
- 每段录屏的建议时长和必须出现的画面。

验收标准：

- 按清单录完可以覆盖评分点。
- 录屏失败时知道需要补录哪一段。

边界条件：

- 不要求今天必须产出视频文件，但必须产出可执行录屏清单。
- 不做剪辑方案细化，节点 4 只保证素材完整。

### 任务 6：最终提交前检查

完成内容：

- 检查仓库不含真实 API Key。
- 检查 `.env` 不被提交。
- 检查新增文档能解释现有代码，而不是脱离代码。
- 检查最终命令清单可执行。

建议命令：

```bash
git status --short
rg -n "sk-|api[_-]?key|secret|password|token" .env* docs apps packages infra scripts package.json
```

验收标准：

- 没有真实密钥。
- 如果命中的是示例字段或普通配置名，必须在最终说明里标注为非真实凭据。

边界条件：

- 不删除用户未确认的本地文件。
- 不为了清理 git 状态而回滚用户已有改动。

## 7. 审批 Agent 验收规则

审批 agent 只判断交付是否满足节点 4，不负责实现。

必须检查：

- P0 六个任务是否都有对应文档或代码证据。
- 是否存在“文档说有，代码/脚本没有”的虚假交付。
- 是否存在真实密钥泄露风险。
- 是否存在节点 4 禁止新增的大功能。
- 测试命令、并发命令、E2E 命令是否写清楚。
- 演示脚本是否可照着录屏。

审批输出格式：

```text
审批结论：通过 / 不通过
P0 阻塞项：
- ...
P1 建议项：
- ...
需要修改 agent 处理的文件：
- ...
复验命令：
- ...
```

只有当 P0 阻塞项为空，才算审批通过。

## 8. 修改 Agent 执行规则

修改 agent 负责按审批结果修正文档、脚本或小范围代码。

执行规则：

- 每轮只处理审批 agent 明确指出的问题。
- 不回滚别人改动。
- 不扩展节点 4 边界。
- 每轮修改后说明改了哪些文件、为什么改、如何验证。
- 如果审批指出的问题需要环境依赖但本地无法验证，必须留下可执行命令和风险说明。

## 9. 完成定义

节点 4 全部完成必须同时满足：

- P0 任务 1-6 全部完成。
- 审批 agent 输出“审批结论：通过”，且 P0 阻塞项为空。
- 本地至少完成 `npm run test` 和 `npm run build`。
- 如果本地环境可用，完成 `npm run check:env`、`npm run redis:ping`、`npm run test:e2e` 和四种并发模式。
- 最终回复中列出已新增/修改文件、验证命令和未验证原因。
