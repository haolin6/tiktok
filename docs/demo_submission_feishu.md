## 4. 项目简介

本项目面向抖音电商直播场景，构建了一个直播竞拍全栈系统。系统使用本地固定素材模拟直播间，重点实现商家发布竞拍、用户实时出价、多档出价与加价器、多端同步、自动延时、成交/流拍/取消终态、订单生成、模拟支付和并发一致性验证。

## 5. 核心功能清单

1. 商家后台发布竞拍：支持配置起拍价、加价幅度、封顶价、竞拍时长和自动延时参数。
2. 竞拍规则编辑：Scheduled 未开始竞拍可修改规则，Running/Sold/Passed/Canceled 不允许修改。
3. 用户直播间实时出价：A/B/C 用户可通过常驻入口进入同一直播间，刷新后保留用户身份；支持下一口价、多档出价和 `+ / -` 加价器。
4. WebSocket 多端同步：当前价、领先者、最近出价、排行榜、成交、流拍、取消、支付和被超越提醒实时同步；直播间展示领先、被超越、最后倒计时和当前排名反馈。
5. 成交与支付闭环：封顶或到期有赢家时生成订单，赢家进入模拟支付页，后台订单同步更新。
6. 并发一致性验证：使用 Redis lock、Redis 幂等 key、MySQL 事务和唯一约束保证并发出价结果一致。

## 6. 端到端使用流程

商家先进入后台发布竞拍，填写商品信息并设置竞拍规则。创建并开始后，用户 A/B/C 通过各自常驻入口进入直播间。用户可按下一口价出价，也可用加价器选择更高档金额；系统要求金额按加价幅度对齐，超过封顶价时按封顶价成交。用户出价后，所有直播间页面实时同步当前价、领先者、最近出价和排行榜。若用户被其他人超过，会收到被超越提醒；若临近结束时发生有效出价，系统会自动延长倒计时。竞拍达到封顶价或到期有赢家时进入成交状态并生成订单，赢家可以进入模拟支付页完成支付。若无人有效出价，竞拍进入流拍状态；若商家取消竞拍，用户端立即同步取消状态并停止倒计时。

## 13. 关键工程难点与解决方案

1. 并发出价一致性  
   竞拍出价天然存在多人同时点击的问题。系统使用 Redis lock 控制关键区，使用 Redis 幂等 key 处理重复请求，并用 MySQL 事务和唯一约束兜底，保证同一时刻只有合法出价生效。多档出价会在锁内基于最新当前价重新校验，避免前端过期金额覆盖最新结果。

2. 多端实时同步  
   用户直播间需要在不同浏览器窗口中实时同步价格、领先者、排行榜、加价器选中金额和终态。系统使用 Socket.IO 按竞拍房间广播事件，并在重连或刷新时通过 snapshot 恢复当前状态；若本地选中金额已低于最新下一口价，前端会自动抬到最新可出金额。

3. 竞拍终态一致性  
   成交、流拍、取消、支付之间不能互相串状态。系统将 Draft、Scheduled、Running、Sold、Passed、Canceled 设计为明确状态机，并补充了取消后不显示错误支付入口、取消后倒计时停止等回归测试。

## 14. 项目亮点 / 创新点

1. 完整直播竞拍闭环：覆盖商家发布、用户多档出价、实时同步、成交订单、模拟支付、流拍和取消。
2. 竞价体验更接近真实直播间：支持 `+ / -` 加价器、领先反馈、被超越提醒、最后倒计时动画和排行榜位置提示。
3. 工程可信度较高：不仅有页面演示，也有 API 测试、E2E 测试、并发脚本和 fanout 证据。
4. AI 协作过程可追溯：保留了方案、测试、演示脚本、AI 使用说明和最终验收日志，展示了 AI 编程从需求拆解到交付验证的完整路径。

## 15. 性能指标 / 压测结果

本机并发脚本已验证 100 个不同用户同时参与同一竞拍时的出价一致性。WebSocket fanout 脚本已验证 100/200 个 Socket.IO 客户端可以收到真实出价触发的广播事件。该结果用于证明本机工程链路可用；线上规模压测不属于当前提交版本范围。
