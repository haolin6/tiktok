## 1. 项目定位

课题名称为「实时竞拍大师」：抖音电商直播竞拍全栈系统设计与实现。

目标是从 0 到 1 构建一个面向直播电商场景的实时竞拍系统，覆盖商品上架、规则配置、实时出价、动态排名、竞拍成交的完整闭环。项目重点不只是做页面，而是体现高并发实时应用中的 WebSocket、分布式锁、状态机、数据一致性和异常兜底能力。

典型场景：直播间中高价值商品，如珠宝、艺术品、二手奢侈品等，通过竞拍完成动态定价。

## 2. 核心业务闭环

必须形成以下主链路：

1. 商家或主播创建竞拍商品。
2. 配置竞拍规则：起拍价、加价幅度、竞拍时长、封顶价、延时机制等。
3. 用户进入直播间查看商品、规则、当前价格、参与人数。
4. 用户实时出价。
5. 系统校验出价合法性并广播最新价格、排名、倒计时和提醒。
6. 若达到封顶价，自动成交。
7. 若临近结束时有人出价，自动延长 10-30 秒。
8. 竞拍结束后生成成交结果和订单。
9. 用户查看结果、模拟支付、查看历史竞拍记录。
10. 主播或商家可查看商品进度、成交结果，并在异常时取消竞拍。

## 3. 功能模块拆解

### 商家/主播端：PC 管理后台

- 竞拍发布：上传商品名称、图片、介绍。
- 规则配置：起拍价、加价幅度、竞拍时长、封顶价、延时机制。
- 商品管理：查看全部竞拍商品状态、进度、成交结果。
- 规则修改：支持修改未开始竞拍的规则。
- 异常处理：支持主播取消异常竞拍。
- 订单管理：成交后自动生成订单，并查看成交详情。

### 用户端：移动端 H5/小程序

- 直播间：可用固定视频或开源库模拟直播画面。
- 竞拍浏览：商品列表、商品详情、竞拍规则、当前出价、参与人数。
- 实时提醒：接收出价提醒。
- 出价参与：手动出价、实时查看排名。
- 关键通知：被超越、竞拍延时、竞拍结束。
- 结果查看：查看成交情况、模拟支付、浏览历史竞拍记录。

## 4. 技术架构要求

宣讲版给出的架构关键词：

- 前端：React + TypeScript，组件化开发，清晰状态管理。
- 后端：优先 Node.js 或 Go，也允许其他语言。
- 通信：RESTful API + WebSocket 双通道。
- 数据库：MySQL 或 PostgreSQL 存储核心业务数据。
- 缓存与高频状态：Redis 应对高频读写。
- 实时通信：WebSocket 长连接，支持直播间/房间级隔离，多直播间互不干扰。
- 工程质量：分层架构合理，注释清晰，可维护、可扩展。

关键技术关键词：

- WebSocket 长连接
- 心跳保活
- 断线重连
- 乐观锁
- 分布式锁
- 防抖节流
- 状态机
- 房间级路由隔离
- 出价幂等性
- 缓存防击穿
- 可观测性与异常告警

## 5. 核心挑战

### 规则复杂度

竞拍规则要求零漏洞：

- 0 元起拍。
- 每次出价必须按固定幅度递增。
- 达到封顶价自动成交。
- 结束前有人出价时自动延时 10-30 秒。
- 主播可以随时取消异常竞拍。

### 实时同步

基础目标是 100+ 用户同时出价，所有用户看到一致的价格、排名和状态。

系统需要做到：

- 出价数据秒级同步。
- 所有人看到的排名一致。
- 倒计时尽量精确，文档强调毫秒级。
- WebSocket 稳定连接，网络波动时自动重连。
- 避免数据延迟、页面卡顿、排名错乱。

### 高并发与一致性

加分目标是单直播间 1000+ 用户在线。

需要重点处理：

- 同一商品的并发出价顺序。
- 重复点击或重试导致的幂等问题。
- Redis 与数据库之间的数据一致性。
- 成交、取消、延时、结束等状态转换的原子性。
- 一笔出价不能被重复扣款或重复计入。

- 后端服务：出价校验、状态机管控。
- 接口网关。
- 前端交互：氛围动画、实时反馈。
- 系统可用性：断连重连、异常兜底。
- 性能与稳定性：缓存防击穿、数据一致性。
- 可观测性：竞拍状态监控、异常告警。

### 技术深度与创新性：25%

重点考察技术选型是否适配高并发直播竞拍场景，以及是否针对实时同步、高并发、WebSocket 不稳定等核心问题做了专门优化。

可体现差异化的方向：

- 房间级 WebSocket 路由隔离。
- 出价幂等性设计。
- 跨端状态同步优化。
- 高并发下的缓存与锁策略。

### AI 使用与落地效果：15%

重点考察：

- 是否合理使用 AI 工具辅助编码、模型调用、数据处理。

- AI 使用流程是否规范、可追溯。
- AI 生成代码占核心代码的比例是否合理。
- AI 代码是否规范、可复用。

注意：文档明确表示 AI 代码贡献率不是越高越好，关键在于关键决策点要有人为把控。
