# 2026-06-06 节点 3 详细 Spec：实时体验和并发证明雏形

## 1. 今日目标

节点 3 的目标是把 2026-06-03 已经跑通的 P0 主链路，从“HTTP 轮询也能演示”升级为“像直播竞拍系统”：

后台发布竞拍 -> 用户进入直播间 -> WebSocket 订阅房间和竞拍 -> A/B/C 或多用户实时出价 -> 所有人立即看到价格、领先者、排行榜和倒计时变化 -> 结束前有效出价触发自动延时 -> 并发脚本证明重复请求和同时出价不会把价格抬乱 -> 最终只生成一个成交订单。

今天的核心不是新增更多页面，而是补齐实时通信、自动延时、Redis 幂等/锁、并发演示和可讲清楚的一致性证据。

## 7. 任务 2 [P0]：WebSocket 服务端和房间订阅

完成内容：

- 新增 `apps/api/src/realtime/realtime-hub.ts` 或同等模块。
- 在 `server.ts` 中把 Socket.IO 挂到同一个 HTTP server 上，端口继续使用 `API_PORT=4000`。
- 支持客户端发送：

  - `room.leave`
  - `auction.subscribe`
  - `bid.place`
- 房间命名：
  - 直播间房间：`room:{roomId}`
  - 竞拍房间：`auction:{auctionId}`
  - 用户私有房间：`user:{userId}`
- `room.join`：

  - 校验 `userId` 存在且 role 是 `bidder` 或允许进入该房间的演示用户。

  - 必须已经成功 `room.join`，否则拒绝订阅。
  - 必须校验 `auction.roomId === socket.data.roomId`，禁止跨直播间订阅。

  - 从 `socket.data.userId` 推导出价用户；没有成功 `room.join` 的 socket 不能出价。

## 8. 任务 3 [P0]：自动延时

完成内容：

- 在有效出价成功后判断是否触发延时：
  - 竞拍状态必须是 `Running`。
  - 当前出价没有触发封顶成交。
  - `endAt - now <= extendThresholdSec`。
- 触发延时时：
  - 更新 `auctions.end_at`。
  - `version + 1`。
  - 写入 `auction_events`：`auction.extended`。
  - 广播 `auction.extended` 和最新 `auction.snapshot`。
- 延时时长：
  - 使用 auction 自身的 `extend_duration_sec`。
  - `newEndAt = max(oldEndAt, now) + extendDurationSec`。
- 前端倒计时必须以服务端最新 snapshot 的 `endAt` 和 `serverTime` 为准更新。

约束：

- 不新增 `Extended` 状态；延时仍属于 `Running`。
- 同一个有效出价最多触发一次延时。
- 达封顶价时优先成交，不再延时。
- 已经 `Sold`、`Passed`、`Canceled` 的竞拍不能延时。
- 延时逻辑必须和出价事务在同一条一致性路径里完成，不能前端自己改倒计时。

测试：

- 使用可控 `end_at` fixture 或注入时钟创建一个剩余 5 秒、`extendThresholdSec=10`、`extendDurationSec=15` 的 Running 竞拍，避免测试依赖真实等待导致 flaky。
- 有效出价后断言：
  - HTTP 或 WS 返回的 snapshot 中 `endAt` 变晚。
  - `auction_events` 有 `auction.extended`。
  - WebSocket 订阅者收到 `auction.extended`。
  - `version` 增加。
- 创建一个剩余 20 秒的 Running 竞拍，有效出价不延时。
- 创建一个达封顶价的出价，断言直接 `Sold`，不写 `auction.extended`。

## 9. 任务 4 [P0]：Redis 幂等和出价锁

完成内容：

- 新增 Redis 配置：
  - `REDIS_HOST`
  - `REDIS_PORT`
  - `REDIS_PASSWORD`
  - `REDIS_DB`，可选，默认 0。
- 新增 Redis client 模块，API 启动时连接，关闭时释放。
- 出价前使用 Redis 锁：
  - key：`auction:{auctionId}:lock`
  - value：随机 token。
  - TTL：2-3 秒。
  - 获取方式：`SET key token NX PX ttlMs`。
  - 释放方式：Lua 校验 token 后删除，避免误删别人的锁。
- 出价幂等 key：
  - key：`auction:{auctionId}:user:{userId}:request:{requestId}`
  - TTL：至少 5 分钟。
  - value：`accepted` / `rejected` / `processing` 和 bidId 或原因。
- 出价执行顺序必须固定：
  1. 完成基础 payload 校验、用户存在和 auction 存在校验。
  2. 先写 Redis 幂等 processing：`SET auction:{auctionId}:user:{userId}:request:{requestId} processing NX PX 300000`。
  3. 如果 SET 失败，读取该 key：
     - 已是 `accepted` 或 `rejected`：直接返回缓存结果，不再次写 bid、不再次抬价。
     - 仍是 `processing`：返回 409 `DUPLICATE_PROCESSING`，不进入抬价逻辑。
  4. processing 写入成功后，再获取 `auction:{auctionId}:lock`。
  5. 获取锁成功后进入 MySQL 事务和状态推进。
  6. 事务成功后把幂等 key 改成最终 `accepted:{bidId}` 或 `rejected:{bidId}:{reason}` 并保留 TTL。
  7. 事务异常时删除 processing key 或写入明确失败态，不能让请求永久卡在 processing。
- 出价成功后可把实时状态写入 Redis：
  - `auction:{auctionId}:state`
  - `auction:{auctionId}:bids` sorted set。
- Redis 不是最终依据；MySQL 的 `uniq_bids_request`、行锁和订单唯一键继续作为兜底。

约束：

- Redis 锁必须包住“读取当前竞拍状态 -> 校验下一口价 -> 写 bid -> 更新 auction -> 延时/成交”的关键路径。
- 获取不到锁时采用短重试：最多 3-5 次，总等待不超过 250ms，每次 25-50ms jitter。
- 短重试后仍获取不到锁时，返回 409 `LOCK_BUSY`，写入 MySQL rejected bid（`reject_reason = lock_busy`）和 `auction_events` 的 `bid.rejected`，并把幂等 key 置为 `rejected:{bidId}:lock_busy`。这类请求不进入价格更新路径。
- Redis 幂等命中时不能再次进入抬价逻辑。
- Redis 失败时不能静默绕过并发保护。开发环境可以显式报错，避免演示时以为有 Redis 但其实没生效。
- MySQL 事务和唯一键不能删除；Redis 只增强一致性和演示说服力。
- 锁释放必须用 Lua 校验 token 后删除，并放在 finally 路径，避免误删别人的锁。

测试：

- 重复同一个 `requestId`：
  - 第一次请求可成功或失败。
  - 第二次请求不能再次改变价格。
  - MySQL 里同一个 `(auction_id, user_id, request_id)` 只有一条 bid。
- 100 个并发请求同时打同一个下一口价：
  - 最多 1 个 accepted。
  - 价格只上涨一次。
  - 失败请求有明确 reject reason。
  - `LOCK_BUSY` 单独计数，同时也计入 rejected 总数。
  - MySQL 中 rejected bid 行数必须和脚本输出的 rejected 总数对得上；其中 `reject_reason = lock_busy` 的数量必须等于脚本输出的 `lockBusy`。
- Redis 锁释放后，下一次合法出价仍能继续。
- 重复请求命中已完成幂等 key 时，不新增第二条 MySQL bid，不新增第二条 rejected event。

## 10. 任务 5 [P0]：并发演示脚本

完成内容：

- 新增脚本，例如 `apps/api/src/scripts/simulate-concurrency.ts`。
- 根目录新增 npm script，例如：
  - `npm run demo:concurrency`
- 脚本流程：
  1. 确保 MySQL 和 Redis 可连接。
  2. 创建或复用 100 个 bidder 测试用户。
  3. 创建一个新商品和新竞拍。
  4. 设置 `startPrice = 99`、`incrementStep = 10`、`ceilingPrice = 109`。
  5. 开始竞拍。
  6. 100 个用户并发提交同一个下一口价 109，requestId 各不相同。
  7. 输出 accepted、rejected、lockBusy、businessRejected、duplicate、durationMs、finalPrice、finalWinnerId、orderCount、event counts。
  8. 查询 MySQL 证明最终只有一个成交订单。
- 脚本输出必须适合直接贴到汇报里，例如：
  - `attempts=100`
  - `accepted=1`
  - `rejected=99`
  - `lockBusy=...`
  - `businessRejected=...`
  - `duplicate=0`
  - `finalStatus=Sold`
  - `finalPrice=109`
  - `orders=1`
  - `acceptedBidRows=1`
  - `rejectedBidRows=99`
  - `rejectedEventRows=99`
  - `durationMs=...`

约束：

- 脚本必须创建独立竞拍，不能污染正在手工演示的竞拍。
- 脚本不能清空表，不能 `TRUNCATE`。
- 测试用户可以用固定前缀 upsert，例如 `load_user_001`。
- 并发请求可以直接调用 service，也可以通过 HTTP 调 API；如果通过 service，仍必须走同一套 Redis 锁和 MySQL 事务。
- 脚本失败时要输出原因和 auctionId，方便查 MySQL。
- 输出口径必须满足：`attempts = accepted + rejected + duplicate`，且 `rejected = lockBusy + businessRejected`。默认 100 个 requestId 各不相同，所以 `duplicate=0`；如果后续增加重复请求模式，重复命中幂等 key 不应新增 MySQL bid 行。

测试：

- `npm run demo:concurrency` 能运行成功。
- 重复运行不会因为用户已存在而失败。
- MySQL 抽查：
  - `orders` 对该 auction 只有 1 条。
  - `bids.accepted = TRUE` 只有 1 条。
  - `bids.accepted = FALSE` 数量等于脚本输出 `rejectedBidRows`。
  - `reject_reason = lock_busy` 数量等于脚本输出 `lockBusy`。
  - `auction_events` 包含 `auction.started`、`bid.accepted`、`auction.sold`、`order.created`，并记录 `bid.rejected` 事件或 rejected bid，数量和脚本输出一致。

  a.id,
  a.status,

  a.current_winner_id,

  SUM(CASE WHEN b.accepted THEN 1 ELSE 0 END) AS accepted_bids,
  SUM(CASE WHEN NOT b.accepted THEN 1 ELSE 0 END) AS rejected_bids,
  SUM(CASE WHEN b.reject_reason = 'lock_busy' THEN 1 ELSE 0 END) AS lock_busy_bids
FROM auctions a
LEFT JOIN orders o ON o.auction_id = a.id
LEFT JOIN bids b ON b.auction_id = a.id

GROUP BY a.id, a.status, a.current_price, a.current_winner_id;
