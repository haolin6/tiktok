# 单人开发排期规划：2026-05-28 至 2026-06-10

适用场景：

- 团队规模：1 人。
- 可用时间有限，需要同时兼顾实验室项目和实习面试准备。
- 节点节奏：每周三、每周六做一次阶段验收。
- 开发截止：2026-06-10。
- 项目演示：2026-06-11 至 2026-06-12。

## 1. 排期原则

### 先闭环，再亮点

本项目的核心不是堆功能，而是把竞拍主链路跑通：

商品发布 -> 规则配置 -> 用户进入直播间 -> 实时出价 -> 排名/倒计时同步 -> 延时/成交/取消 -> 订单生成。

只有主链路稳定后，才做动画氛围、并发证明和更完整的文档。

### 用固定视频模拟直播

直播推流不做第一阶段目标。用户端直接使用固定视频或本地素材模拟直播间，把时间留给竞拍规则、WebSocket 同步和移动端体验。

### 每个节点都必须可演示

每周三和每周六不是“写了一些代码”的节点，而是“能打开页面、能跑命令、能证明一段链路”的节点。

### 每个节点都保留砍范围规则

如果某个节点延期，优先砍 P1/P2，不砍 P0 主链路。砍范围顺序：

1. 真实直播流。
2. 复杂数据看板。
3. 接口网关、鉴权、限流。
4. 多直播间。
5. 高级动画和声音。
6. 1000 在线模拟。

保留：

- 商品发布。
- 用户出价。
- WebSocket 实时同步。
- 竞拍状态机。
- 成交订单。
- 一份讲得清楚的技术方案。

## 2. 时间假设

为了适配单人低可用时间，按保守工作量估算：

- 工作日：每天 1-2 小时碎片时间。
- 周六：4-6 小时相对集中的开发时间。
- 周日：只作为缓冲，不强排硬任务。
- 每个节点前一天晚上只做集成和验收，不再开大功能。

如果实际时间更少，直接走“保底路线”；如果时间更多，再补“加分路线”。

## 4. 节点 0：2026-05-28 周四

目标：完成范围冻结和项目路线确认。

必须完成：

- 明确不做真实直播流。

- `npm run check:env`

## 5. 节点 1：2026-05-30 周六

主题：项目骨架和后端基础链路。

目标：后台还不需要好看，但后端必须能表达竞拍业务。

必须完成：

1. 工程骨架
   - `apps/web`：React + TypeScript + Vite。
   - `apps/api`：Node.js + TypeScript 后端。
   - `packages/shared`：共享类型、状态枚举、事件协议。

2. 数据库
   - 建表迁移：users、products、auction_rooms、auctions、bids、orders、auction_events。
   - Seed 数据：至少 1 个主播、3 个用户、1 个直播间。

3. 后端 HTTP API
   - 创建商品。
   - 创建竞拍。
   - 查询竞拍列表。
   - 查询竞拍详情/快照。
   - 取消竞拍。

4. 状态机基础
   - Draft / Scheduled / Running / Sold / Passed / Canceled。
   - 只允许合法状态转换。

建议完成：

- 简单 Swagger/OpenAPI 或接口文档。
- 后端结构化日志。
- 管理端最简页面能看到竞拍列表。

验收方法：

- `npm run dev` 或分别启动 web/api。
- 通过 HTTP 请求创建商品和竞拍。
- MySQL 中能看到商品和竞拍记录。
- 取消竞拍后状态正确变化。

如果来不及：

- 先不做漂亮后台。
- 先不做订单。
- 先不做 WebSocket。
- 但必须完成数据表和竞拍创建 API。

节点产物：

- 可运行项目骨架。
- 后端 API 初版。
- 数据库迁移/初始化脚本。

## 6. 节点 2：2026-06-03 周三

主题：P0 主链路打通。

目标：从后台创建竞拍，到用户端进入直播间并出价，完整跑一遍。

必须完成：

1. 用户端直播间
   - 固定视频背景。
   - 商品卡片。
   - 当前价。
   - 倒计时。
   - 出价按钮。

2. 出价 API
   - 出价金额校验。
   - 加价幅度校验。
   - 竞拍状态校验。
   - 出价记录落 MySQL。
   - 当前价格和领先者更新。

3. 成交规则
   - 达封顶价自动成交。
   - 到结束时间后自动成交或流拍。
   - 成交后生成订单。

4. 管理端
   - 创建竞拍表单。
   - 竞拍列表。
   - 订单列表。

5. 用户端结果
   - 成交结果页。
   - 模拟支付按钮。
   - 我的历史竞拍/订单入口。

建议完成：

- 前端基础样式。
- 移动端适配。
- 出价成功/失败 toast。

验收方法：

1. 管理端创建一个 30 秒竞拍。
2. 用户端打开直播间。
3. 用户连续出价。
4. 价格正确上涨。
5. 倒计时结束后生成订单。
6. 管理端订单列表能看到成交订单。

如果来不及：

- 先不做 WebSocket 实时广播，用轮询/手动刷新临时保主链路。
- 先不做排行榜动画。
- 先不做历史记录复杂筛选。
- 但必须能从发布走到订单。

节点产物：

- 第一个可演示 MVP。
- P0 主链路初步闭环。

## 7. 节点 3：2026-06-06 周六

主题：实时体验和技术亮点。

目标：把 MVP 从“能跑”升级为“像直播竞拍系统”。

必须完成：

1. WebSocket 房间
   - 用户进入 `roomId` 房间。
   - 订阅 `auctionId`。
   - 服务端广播竞拍快照。
   - 重连后重新拉取快照。

2. 实时事件
   - `auction.snapshot`
   - `bid.accepted`
   - `bid.rejected`
   - `ranking.updated`
   - `auction.extended`
   - `auction.sold`
   - `auction.canceled`

3. 自动延时
   - 结束前 N 秒有效出价触发延时。
   - 所有用户收到延时提醒。

4. 幂等和并发
   - `bidRequestId`。
   - Redis 幂等 key。
   - Redis 锁或原子脚本。
   - 重复请求不会重复计价。

5. 并发演示
   - 脚本模拟 100 个用户出价。
   - 输出成功、失败、最终价格、最终赢家、耗时。

建议完成：

- 最后 10 秒倒计时动效。
- 领先/被超越提醒。
- 排行榜基础动画。
- 管理端监控卡片：在线人数、出价次数、延时次数、异常拒绝次数。

验收方法：

1. 同时打开两个用户端页面。
2. 用户 A 出价，用户 B 立即看到当前价变化。
3. 用户 B 出价超过 A，用户 A 收到被超越提醒。
4. 结束前出价触发延时。
5. 运行 100 人模拟脚本，最终只生成一个成交订单。

如果来不及：

- 先做 WebSocket 快照广播，不做完整事件补偿。
- 先做基础排行榜，不做动画。
- 并发脚本可以只测 HTTP 出价，不一定测 WebSocket 在线连接。
- 但必须有幂等和一致性证明。

节点产物：

- 可展示的实时竞拍体验。
- 可讲的技术亮点。
- 可运行的并发测试脚本。

## 12. 风险控制

### 最大风险：主链路不闭环

控制方式：

- 节点 2 前必须跑通“发布 -> 出价 -> 成交 -> 订单”。
- WebSocket 可以晚一点，但订单闭环不能晚。

### 第二风险：时间被实验室和面试吃掉

控制方式：

- 每个节点只有一个核心目标。
- 平日只做小任务。
- 周六做集成。

### 第三风险：前端和后端同时半成品

控制方式：

- 先用后端 API 和简单页面打通。
- 样式精修放到节点 3。

### 第四风险：技术亮点讲不清

控制方式：

- 从节点 1 开始记录 AI 使用和关键决策。
- 节点 3 必须有并发脚本。
- 节点 4 只整理，不再补大功能。

## 13. 推荐最终取舍

保底交付：

- 单直播间。
- 单商品当前竞拍。
- PC 后台创建和查看。
- H5 用户端出价。
- WebSocket 实时同步。
- 成交订单。
- 固定视频模拟直播。

主打亮点：

- 抖音直播竞拍氛围还原。
- Redis 锁 + 幂等出价。
- 100 人并发出价模拟。
- AI 辅助开发过程可追溯。

明确不做：

- 真实直播推流。
- 真实支付。
- 复杂云部署。
- 千万级压测。
- 大而全后台系统。
