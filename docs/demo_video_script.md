# 三分钟成果演示视频脚本

## 视频目标

三分钟视频展示完整主链路和 2026-06-10 交付完善亮点：

```text
验收者快速路径 -> 后台发布竞拍 -> Scheduled 规则编辑 -> 双端/多端直播间多档出价 -> 加价器/被超越提醒/自动延时 -> 成交订单 -> 模拟支付 -> 流拍结果 -> 并发与 fanout 证据 -> AI 协作边界
```

本地地址：

- Web：`http://127.0.0.1:3000`
- API：`http://127.0.0.1:4000`

录制前执行：

```bash
npm run check:env
npm run db:migrate
npm run db:seed
npm run dev
```

录制前准备：

1. 将后台发布页表单预填好，主链路竞拍时长建议设为 `90` 秒，避免中途切换页面后竞拍过早到期。
2. 准备一个 `Scheduled` 候选竞拍，用来演示规则编辑；不要用正在出价的主链路竞拍做编辑演示。
3. 准备一个流拍候选竞拍：创建时长 `5` 秒的竞拍并开始，不出价，等待到期后打开 `/live/:roomId?auctionId=<id>` 触发快照结算，确认直播间显示 `竞拍已流拍，无人成交`。
4. 如需稳定展示自动延时，单独准备一个临近结束的 `Running` 候选竞拍，在倒计时进入触发阈值内再出价；主链路视频里至少展示延时规则和一次延时提示。
5. 并发和 fanout 终端命令可在录制前运行并保留输出，视频里展示关键结果，避免三分钟内等待脚本执行。
6. 提前确认或收藏 A/B/C 三个常驻用户入口，例如 `/live/:roomId?userId=<a-id>`、`/live/:roomId?userId=<b-id>`、`/live/:roomId?userId=<c-id>`；发布成功面板也会显示这些入口。若只打开两个页面，在线数应显示 2 个已连接用户。

浏览器窗口：

- 窗口 1：后台 `/admin/auctions/new`、`/admin/auctions` 和 `/admin/auctions/:id/edit`。
- 窗口 2：`/live/:roomId?userId=<a-id>`，常驻 `竞拍用户 A`。
- 窗口 3：`/live/:roomId?userId=<b-id>`，常驻 `竞拍用户 B`；需要三用户演示时再开 `userId=<c-id>`。
- 窗口 4：终端，保留并发脚本和 fanout 脚本输出。

## 0:00-0:15 项目定位

画面：README 顶部的 `快速验收路径`。

口播：

```text
这是实时竞拍大师，一个抖音电商直播竞拍全栈系统。验收者可以先看三分钟视频，再按 README 的本地路径和验收命令复现。当前版本完成后台发布、规则编辑、实时出价、WebSocket 同步、成交订单、模拟支付、流拍展示和并发一致性验证。直播画面使用本地素材模拟，支付为模拟支付。
```

## 0:15-0:40 后台发布竞拍

页面：`/admin/auctions/new`

操作：

1. 填写商品名称、图片和介绍。
2. 设置起拍价 `99`、加价幅度 `10`、封顶价 `129`、时长 `90` 秒。
3. 设置自动延时参数，例如延时触发阈值 `10` 秒、每次延长 `15` 秒；原始要求是结束前出价自动延长 `10-30` 秒，当前实现支持配置。
4. 点击 `创建并开始`。

画面结果：

- 页面显示竞拍 ID。
- 状态进入 `Running`。
- 出现 `进入直播间` 链接，直接进入 `/live/:roomId?auctionId=<id>`；同时出现 A/B/C 常驻用户入口，便于三个用户页面保留各自身份并跟随当前竞拍。

口播：

```text
主播在后台配置商品和竞拍规则，包含起拍价、加价幅度、封顶价和结束前自动延时参数。创建并开始后，竞拍进入 Running 状态，用户可以进入同一个直播间参与出价。
```

## 0:40-1:00 Scheduled 规则编辑

页面：`/admin/auctions` -> `/admin/auctions/:id/edit`

操作：

1. 打开录制前准备好的 `Scheduled` 候选竞拍。
2. 点击 `编辑`。
3. 修改起拍价、加价幅度、封顶价、时长和自动延时参数。
4. 保存后回到列表。

画面结果：

- 编辑页只允许修改未开始竞拍。
- 列表中展示更新后的规则。
- `Running`、`Sold`、`Passed`、`Canceled` 终态不会提供规则编辑入口。

口播：

```text
未开始竞拍可以在后台修改规则，这补齐了商家开拍前调整价格和延时策略的场景。已经开始或结束的竞拍不会再允许改规则，避免破坏状态机一致性。
```

## 1:00-1:35 双端实时出价

页面：两个常驻用户窗口并排，例如 `/live/:roomId?userId=<a-id>` 和 `/live/:roomId?userId=<b-id>`。它们会跟随当前 `Running` 竞拍，不需要手动刷新。

操作：

1. 确认左侧是 `竞拍用户 A`，右侧是 `竞拍用户 B`。
2. 用户 A 出价。
3. 用户 A 点击 `+` 选择更高档金额并再次出价，展示“当前您已是最高价”。
4. 用户 B 点击 `+` 选择更高档金额后超过 A。
5. 若要稳定展示自动延时，切到录制前准备的临近结束候选竞拍，在触发阈值内出价。

画面结果：

- 两个窗口连接状态为 `connected`。
- 当前价从 `¥99.00` 到 `¥109.00`，再通过加价器多档提升。
- 领先者、最近出价和排行榜在两个窗口同步变化。
- 加价器显示我的出价、加价幅度和封顶价；当前用户在排行榜中高亮。
- 用户 A 被用户 B 超过后，A 窗口显示 `已被超越` 提醒。
- 倒计时或规则区域能看到自动延时规则；临近结束候选竞拍能展示延时提示。

口播：

```text
直播间通过 Socket.IO 做房间级广播。用户既可以按下一口价出价，也可以用加价器选择更高档金额；金额必须按加价幅度对齐，超过封顶价会按封顶价成交。一个用户出价后，另一个窗口无需刷新即可看到当前价、领先者、最近出价和排行榜更新。被超过的用户会收到 user.outbid 提醒，临近结束时的有效出价会触发自动延时，避免最后一秒截拍。
```

## 1:35-2:00 成交订单和模拟支付

页面：常驻用户窗口、`/pay/:orderId`、`/admin/orders`

操作：

1. 在 A 或 B 页面继续出价到封顶价 `129`。
2. 直播间显示成交。
3. 赢家页面点击 `去支付`。
4. 支付页点击 `模拟支付`。
5. 打开后台订单列表。

画面结果：

- 竞拍状态为 `Sold`。
- 支付页状态从 `pending_payment` 变为 `paid`。
- 后台订单列表显示买家、金额和支付状态。

口播：

```text
达到封顶价后系统自动成交并生成订单。模拟支付会更新 MySQL 中的订单状态，并通过 WebSocket 广播 order.paid。
```

## 2:00-2:20 流拍或取消终态

页面：`/admin/auctions`、`/live/:roomId?auctionId=<id>`

操作：

1. 打开录制前准备好的流拍候选竞拍直播间 `/live/:roomId?auctionId=<passed-candidate-id>`。
2. 如果页面还没有进入终态，刷新一次直播间；读取 `/api/auctions/:id/snapshot` 会触发到期结算。
3. 三分钟内优先展示流拍；如时间允许，再演示取消一个未结束竞拍。

画面结果：

- 流拍竞拍显示 `Passed`。
- 直播间展示 `已流拍` 终态，不再提供支付入口。
- 取消竞拍显示 `Canceled`，直播间同步取消原因。

口播：

```text
没有满足成交条件的竞拍会进入 Passed 流拍终态，前端会显示明确结果而不是停在倒计时结束。管理端取消也会同步到直播间，终态竞拍不会再次进入出价或支付流程。
```

## 2:20-2:45 并发一致性和 fanout

页面：终端。

命令：

```bash
npm run demo:concurrency -- --mode=unique
npm run demo:realtime-fanout -- --clients=200
```

画面保留：

```text
attempts=100
accepted=1
rejected=99
finalStatus=Sold
orderCount=1
acceptedBidRows=1
rejectedBidRows=99
clients=200
broadcastReceived=200
broadcastFailures=0
```

口播：

```text
并发脚本模拟 100 个用户同时抢同一口价，最终只有一个请求成功，只有一个订单生成。Redis lock 控制并发关键区，Redis 幂等 key 处理重复请求，MySQL 唯一约束兜底订单唯一性。fanout 脚本用真实 HTTP 出价触发 Socket.IO 广播，本机已验证 200 个客户端收到同一竞拍事件。
```
